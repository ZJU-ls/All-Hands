## 0 · TL;DR

- **聊天 ≠ 任务**。聊天是流式对话(用户在线守着),任务是 fire-and-forget(用户发出去就关窗,回头看结果 + 制品)
- Task 是一等领域对象:`id / title / goal / dod / assignee / status / run_id* / artifact_ids* / created_by / source(user/lead/trigger)`
- 状态机:`queued → running → (needs_input | needs_approval) ↔ running → completed | failed | cancelled`
- UI:cockpit 顶栏 "任务" 计数 + `/tasks` 收件箱 / 详情页。用户关掉页面后任务继续跑,回来能看 **结果 + 产出的全部 artifacts**
- Lead Agent 通过 `tasks.create` Meta Tool 把"我现在派出去干的事"变成 Task,而不是仅仅嵌套一个 dispatch
- Trigger 的 `dispatch_employee` action 内部统一走 Task(原 run 直接路径**保留**,但新建议走 Task 以获得可追溯性)

## 1 · 问题陈述

今天聊天主导一切:用户要让员工干活 → 必须**在对话里**盯着。一旦关掉 `/chat/{id}`,没有地方看"我之前让 writer 写的 release note 好了没?"。

"一个人驱动 AI Team" 要求的**核心产品动作**:

1. 用户提出目标("下周的发布日志整理出来")→ 分派给某员工 → **关闭标签页走人**
2. 员工干到一半卡了 → 需要用户补信息 / 审批 → 用户回来被通知 → 5s 回答 → 继续
3. 员工做完 → 输出 markdown / 图 / 代码 → 存进制品区 → 用户打开看
4. 用户永远不需要问"现在跑到哪了",收件箱能看一切

Chat 是**同步通道**。Task 是**异步通道**。两者并存,**不互替**。

## 2 · 原则

### 2.1 Task 与 Chat 正交

- Chat = 实时双向对话 · 需要用户在线 · 出 token 流
- Task = 异步工作单元 · 用户离线 · 出结果 + artifact
- 关联:Task 执行过程中如果遇到 `needs_input`,自动**开一条**相关 conversation 让用户回答(带 task_id 链接)
- 关联:用户在 chat 里对 Lead 说"帮我整理 release note",Lead **不应**在 chat 内部长跑 —— 应该 `tasks.create` 派出去,chat 里回"已派给 writer,任务 id = T-abc · 完成后通知你",随后关闭注意力

### 2.2 状态机严格

```
                ┌──────────────────┐
                │     queued       │
                └────────┬─────────┘
                         │ assigned & started
                         ▼
                ┌──────────────────┐
            ┌──▶│     running      │──┐
            │   └────┬───────┬─────┘  │
            │        │       │        │
  user answer│       │       │user approve/deny
            │        ▼       ▼        │
            │ ┌─────────┐ ┌──────────┐│
            └─│needs_   │ │needs_    │┘
              │input    │ │approval  │
              └─────────┘ └──────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
     completed        failed        cancelled
```

终态:`completed / failed / cancelled`。进入终态后 `result_summary` + `artifact_ids` 必须填齐。

### 2.3 每个 Task 都有 DoD

Task.dod = "我怎么知道这事儿做完了"。用户创建时必填 · Lead 创建时由 Lead 写并**显式呈现给用户**(聊天回"我给它 DoD 是 X,你看行不?")。DoD 是员工自我判定"done"的依据。缺 DoD 的 Task **不得** 进 `running`。

### 2.4 关窗不杀任务

- Task 的 run 跑在后台 · 不依赖任何 SSE 连接
- 前端关掉 / 重连 / 刷页 → 后端 run 继续
- 重连回来 → 从 events 表拿 task 状态 + run 快照 · 追上

### 2.5 Tool First

用户在 UI 能做的每件事都有同名 Meta Tool:
- `tasks.create` / `tasks.list` / `tasks.get` / `tasks.cancel` / `tasks.answer_input` / `tasks.approve`

## 3 · 数据模型

### 3.1 `Task`(`core/task.py`)

```python
class TaskStatus(str, Enum):
    QUEUED = "queued"
    RUNNING = "running"
    NEEDS_INPUT = "needs_input"
    NEEDS_APPROVAL = "needs_approval"
    COMPLETED = "completed"
    FAILED = "failed"
    CANCELLED = "cancelled"

class TaskSource(str, Enum):
    USER = "user"              # /tasks 新建
    LEAD = "lead"              # Lead Meta Tool 派
    TRIGGER = "trigger"        # trigger action 派
    EMPLOYEE = "employee"      # 员工 dispatch 一个子 Task(深度 ≤ MAX_DISPATCH_DEPTH)

class Task(BaseModel):
    id: str                     # T-<ulid>
    title: str                  # 短 · 显示用("整理下周 release note")
    goal: str                   # 长 · markdown · 任务说明
    dod: str                    # 验收标准 · markdown

    assignee_id: str            # employee_id
    status: TaskStatus

    source: TaskSource
    created_by: str             # user_id / "lead_agent" / trigger_id / parent task_id
    parent_task_id: str | None  # employee 派子 task 时填

    run_ids: list[str]          # 关联所有 run(retry 会多 run 一条)
    artifact_ids: list[str]     # 产出制品
    conversation_id: str | None # 若为了 needs_input 开过 conversation

    result_summary: str | None  # 完成时填 · markdown · 供 cockpit / 收件箱预览
    error_summary: str | None   # 失败时填

    # 需要人介入的
    pending_input_question: str | None       # NEEDS_INPUT 时填
    pending_approval_payload: dict | None    # NEEDS_APPROVAL 时 (tool_id, args, scope, summary)

    # 预算
    token_budget: int | None    # 超过则自动 cancel(None = 无限)
    tokens_used: int

    created_at: datetime
    updated_at: datetime
    completed_at: datetime | None
```

### 3.2 迁移 · `0007_add_tasks.py`

```sql
CREATE TABLE tasks (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    goal TEXT NOT NULL,
    dod TEXT NOT NULL,
    assignee_id TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'queued',
    source TEXT NOT NULL,
    created_by TEXT NOT NULL,
    parent_task_id TEXT REFERENCES tasks(id) ON DELETE SET NULL,
    run_ids JSONB NOT NULL DEFAULT '[]',
    artifact_ids JSONB NOT NULL DEFAULT '[]',
    conversation_id TEXT,
    result_summary TEXT,
    error_summary TEXT,
    pending_input_question TEXT,
    pending_approval_payload JSONB,
    token_budget INTEGER,
    tokens_used INTEGER NOT NULL DEFAULT 0,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    completed_at TIMESTAMPTZ,
    workspace_id TEXT NOT NULL
);
CREATE INDEX idx_tasks_status ON tasks(workspace_id, status, updated_at DESC);
CREATE INDEX idx_tasks_assignee ON tasks(workspace_id, assignee_id, status);
```

## 4 · 执行模型

### 4.1 `TaskExecutor`(`execution/tasks/executor.py`)

- 订阅 DB 里 `status=queued` 的 task · 拉起 run
- 一个 task = 一个 agent run · run 的 `task_id` 字段已存
- run 结束 → task 根据 run.status 进 `completed` / `failed`
- run 中途调 `tasks.request_input(question)` 或 `tasks.request_approval(tool, args)` → task 转 `needs_input` / `needs_approval` · 发通知 · run **挂起**(保存 checkpoint · 见 § 4.4)
- 用户 answer / approve → run 恢复继续

### 4.2 Employee 侧新 tool(自主提问/申请)

让员工能"伸手向外要补充":

```python
Tool(id="allhands.meta.task.request_input", scope=WRITE, requires_confirmation=False,
     description="""Pause current task and ask the user a clarifying question.
Use SPARINGLY — only when you genuinely cannot proceed without the answer.
Do NOT use this for optional polish questions; make a reasonable assumption and note it in result.

Arguments: question (plain-text, one sentence preferred).
After this tool returns, the run is paused. When the user answers, you'll receive their reply
as the next user turn, and you should continue the task.""")

Tool(id="allhands.meta.task.request_approval", scope=WRITE, requires_confirmation=False,
     description="""Pause current task and ask the user to approve a specific action (tool_id + args summary).
This is for side-effectful operations whose risk is outside the normal Gate scope — e.g. publishing,
sending external notifications, large deletes. For standard WRITE/IRREVERSIBLE tools, the Gate
already handles approval; don't wrap those in request_approval.""")
```

两者只在 **当前 run 关联一个 task** 时才可用(否则报错)。未绑定 task 的 run(例如 chat 内的单轮答复)**不能**暂停。

### 4.3 Task 通知(→ Inbox)

Task 进入 / 离开终态 / 进入 needs_* 状态时,emit 事件:

- `task.created` / `task.started` / `task.needs_input` / `task.needs_approval` / `task.answered` / `task.completed` / `task.failed` / `task.cancelled`

这些事件复用 [triggers.md § 4.2](./2026-04-18-triggers.md#42) 的 EventBus + events 表。Cockpit 活动流和 `/tasks` inbox 都订阅。

可选(v0 不强)· 浏览器 Notification API:用户允许后,`task.needs_input` / `task.completed` 弹系统通知。

### 4.4 Run 挂起 · resume

LangGraph 原生 checkpoint 机制(`MemorySaver` / `SqliteSaver`):
- run 进 needs_input → checkpoint 保存 · 活跃 run 队列移除
- 用户 answer → task_service 把答复注入 LangGraph state 的 human message → resume
- checkpoint 存在 DB(v0 用 SQLite 存,与主 DB 同实例)

若执行端当前用 `create_react_agent` 无 checkpoint,本 spec 要求改成带 checkpointer。LangGraph 官方一行配置。

## 5 · Meta Tools

| Tool id | scope | 语义 |
|---|---|---|
| `allhands.meta.tasks.create` | WRITE(+confirmation on create by Lead) | 建新 task · params: title, goal, dod, assignee_id, token_budget? |
| `allhands.meta.tasks.list` | READ | 过滤列 · params: status?, assignee_id?, limit? |
| `allhands.meta.tasks.get` | READ | 详情 · 含 run_ids / artifact_ids / result_summary |
| `allhands.meta.tasks.cancel` | IRREVERSIBLE | 取消 task(含正在跑的 run) |
| `allhands.meta.tasks.answer_input` | WRITE | 给 needs_input 的 task 回答一句 · 触发 resume |
| `allhands.meta.tasks.approve` | WRITE | 批准 needs_approval(或 deny)· params: task_id, decision, note? |
| `allhands.meta.tasks.add_artifact` | WRITE | 把新 artifact 附到 task(有些场景 agent 自己 pin) |

Lead prompt 补丁(`lead_agent.md`):

> When the user asks you to do something that will take more than one of your turns (writing, research, multi-step coordination), **call `tasks.create`** instead of trying to do it inline. Tell the user "I've created task T-xxx for <employee>; I'll let you know when it's done." Then **stop talking**. Don't keep yapping progress updates into the chat.

### 5.1 Task 创建时的 DoD 协商

Lead 调 `tasks.create` 时 **必须** 提供 `dod`。提示补丁:

> Always write a 2-4 bullet DoD. Minimum: "output format", "what must be included", "what definitely must NOT be included", "acceptance signal". If you can't write the DoD, you don't understand the request — ask the user first.

## 7 · 前端

### 7.1 路由

- `/tasks` — 收件箱(默认看非终态)+ 所有筛选
- `/tasks/{id}` — 详情:title · status · DoD · artifacts · 最近 run 流水 · 如 needs_input 有答题框
- cockpit 顶部 KPI 多一格:`N 任务在跑` + 状态色点(有 needs_input → 黄)
- cockpit 右栏快速操作加 "+ 新任务"

### 7.2 新任务 drawer

字段:title(单行)· assignee(员工下拉)· goal(长文本)· DoD(长文本)· token_budget(数字 · 可空) · 草稿保存

**重要 UX**:drawer 顶有 "让 Lead 帮我完善 DoD" 按钮(可选)· 点了就以 "title + goal 草稿" 调一次 Lead 简短对话 → Lead 出 DoD 候选 → 用户改 → 提交。这一条非常符合"一个人驱动 AI"。

### 7.3 详情页

- 顶部状态 + cancel 按钮
- DoD 原文
- 下方 tab:`Run 流水`(events 时间线)/ `Artifacts`(卡片网格,用 `allhands.render` 组件预览)/ `对话`(若开了 conversation_id 给直接链)
- 若 `needs_input`:顶部出高亮条 + 回答框
- 若 `needs_approval`:高亮条 + Approve / Deny 按钮 + 原因输入

### 7.4 视觉(CLAUDE.md § 3.5)

- 状态不用 icon · 用点 + 文字:`● queued` `● running` `● needs input` `● done`
- needs_input / needs_approval 用 `#d97706`(warn)色
- 详情页 run 流水用左侧 2px 色条,不用阴影

活样本 mock 见 brainstorm session `13-task-inbox.html`(autopilot 起工前自行补,或忽略)。

## 10 · 开放问题 · Decision defaults

1. **Q**: chat 里用户说"帮我写一份 release note"怎么识别"该派 task 还是直接答"?
   **Default**: Lead prompt 教,不做 heuristic。Lead 判断 > 1 个 assistant turn 能完成 → tasks.create;≤ 1 turn → 直接 inline。偏差由用户纠正。

2. **Q**: 一个 Task 可以有多 assignee 吗?
   **Default**: v0 单 assignee。多员工协作走"父 task 派子 task(parent_task_id)"。

3. **Q**: 取消正在跑的 run 时,已经写入制品的那些怎么办?
   **Default**: 保留。cancel 只停 run,不回滚 artifact。task.result_summary 写"cancelled at step X"。

4. **Q**: `needs_input` 通知用户的渠道?
   **Default**: v0 · cockpit 顶栏 + `/tasks` inbox 高亮 + 浏览器 tab title 前缀 `(N!)`。v1 加 Notification / email / push。

5. **Q**: Lead 本身能"接 task"吗?
   **Default**: 能。`assignee_id = <lead employee id>` 合法。理由:用户可以说"Lead,你自己整理一下"。但 Lead 对自己派 task 要谨慎(prompt 里提醒"你不是所有事都要自己做")。
