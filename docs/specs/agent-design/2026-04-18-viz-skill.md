## 0 · TL;DR

- 一组结构化渲染 render tool,打包成内置 skill `allhands.render`
- 员工 `skill_ids` 加 `allhands.render` → 自动拿到 10+ render tool(`allhands.render.table` / `.kv` / `.timeline` / ...)
- Skill 的 `prompt_fragment` 教 Agent:**结构化信息优先用 viz.*,别老写 markdown 列表**
- 渲染走现有 SSE `event: render` 管道 → 前端 component-registry 查表

---

## 1 · 问题陈述

今天架构 § L5.8 只列了 5 个 render tool(`markdown` / `employee_list` / `employee_detail` / `confirmation_summary` / `trace_link`),并且只有 `MarkdownCard.tsx` 真的建了。实际对话里 Agent 经常面对:

- "帮我对比 3 个方案" → 应该是 **表格** 或 **cards**,不是 markdown 列表
- "总结这次变更" → 应该是 **diff** + **callout**,不是粘贴一大段
- "给个进度" → 应该是 **timeline** / **steps**,不是"我完成了 1、2、正在做 3"

缺一个**完备的可视化调色盘** + 一套**让 Agent 知道何时用哪个**的机制。

---

## 2 · 原则

### 2.1 Skill 打包 = 能力即插即用

`allhands.render` 是一个 **v1 内置 skill**,包含:
- 若干 render tool
- 一个 `prompt_fragment` 说明"什么场景用哪个 tool"
- 版本号(skill 自己可升级,不影响引用它的员工 tool_ids)

员工 `skill_ids: ["allhands.render"]` → `expand_skills_to_tools()` 把这批 tool 和 prompt 都挂进去。

### 2.2 Render Tool 永不触达外部

每个 `allhands.render.*` 工具:
- `scope = READ`
- `requires_confirmation = False`
- 只做一件事:**把结构化数据 → RenderPayload**
- 不改数据库、不动文件、不调 API

### 2.3 渐进实现

v0 先 10 个高频组件,v1 再补 chart / kanban / form / tree / collab-specific。

## 4 · Schema

每个 tool 的 input schema(Pydantic)放在对应 Python 文件里。统一返回 `RenderPayload`:

```python
# 已定义 product/04-architecture.md § L4.4
class RenderPayload(BaseModel):
    component: str             # e.g. "Viz.Table"
    props: dict                # 组件 props(见上表)
    interactions: list[InteractionSpec] = []
```

**示例 · `allhands.render.table`**:

```python
# backend/src/allhands/execution/tools/render/table.py
TABLE_TOOL = Tool(
    id="allhands.render.table",
    kind=ToolKind.RENDER,
    name="render_table",
    description="Render structured rows into a sortable table. "
                "Use when you need to compare multiple items by multiple attributes.",
    input_schema={
        "type": "object",
        "properties": {
            "columns": {"type": "array", "items": {...}},
            "rows": {"type": "array", "items": {"type": "object"}},
            "caption": {"type": "string"},
        },
        "required": ["columns", "rows"],
    },
    output_schema=RenderPayload.model_json_schema(),
    scope=ToolScope.READ,
    requires_confirmation=False,
)

async def executor(columns, rows, caption=None) -> RenderPayload:
    return RenderPayload(
        component="Viz.Table",
        props={"columns": columns, "rows": rows, "caption": caption},
    )
```

## 5 · Skill 定义

### 5.1 文件位置

```
backend/skills/builtin/render/
├── SKILL.yaml
└── prompts/
    └── guidance.md        ← 长 prompt_fragment
```

### 5.2 `SKILL.yaml`

```yaml
id: allhands.render
name: 可视化渲染
description: 结构化展示(表格、时间线、diff、代码、卡片、提示等)
version: 1.0.0
builtin: true

tool_ids:
  - allhands.render.markdown
  - allhands.render.table
  - allhands.render.kv
  - allhands.render.cards
  - allhands.render.timeline
  - allhands.render.steps
  - allhands.render.code
  - allhands.render.diff
  - allhands.render.callout
  - allhands.render.link_card

prompt_fragment_file: prompts/guidance.md
```

### 5.3 `guidance.md`(教 Agent 用)

```markdown
# 可视化渲染指南

当你要向用户展示信息时,**优先使用 allhands.render.* 工具**,而不是纯文字 / markdown 列表。

## 判断表

| 你想展示的内容 | 用这个工具 |
|---|---|
| 多条记录的多个属性对比 | `allhands.render.table` |
| 单个对象的详情 | `allhands.render.kv` |
| 3-6 个并列方案 / 选项 | `allhands.render.cards` |
| 时间顺序的事件 / 计划 | `allhands.render.timeline` |
| 固定顺序的步骤 / wizard | `allhands.render.steps` |
| 代码片段 | `allhands.render.code` |
| 代码或文本变更 | `allhands.render.diff` |
| 重要提示 / 警告 / 成功 | `allhands.render.callout` |
| 外部链接推荐 | `allhands.render.link_card` |
| 长篇说明(>500 字) | `allhands.render.markdown` |

## 何时不用

- 单行简短回复 → 直接说,不用 render
- 数据超过 100 行 → 用 artifact.create 存成 data 制品,再 artifact.render
- 不确定用哪个 → 先给 markdown,后续迭代
```

### 6.3 视觉规范约束

- **严格遵守** `product/03-visual-design.md` · Linear Precise:
  - 颜色密度 ≤ 3(不含语义色)
  - 禁止 icon 库 —— table sort arrows 用 Unicode 或 1-line SVG

  - 动效走 `--dur-*` token
- **每个组件** 进 `web/app/design-lab/page.tsx` 作为活样本验证

## 10.5 · 参考源码(动手前必读)

> 规则见 [`docs/claude/reference-sources.md`](../../claude/reference-sources.md)。本 spec 有两类参考点:**Tool 机制**(后端)+ **prompt 教学法**(skill 的 guidance.md 怎么写)。

| 本 spec 涉及 | 对标 ref-src-claude 入口 | 抽什么 |
|---|---|---|
| **§ 3 Tool 定义模式 · input/output schema** | `ref-src-claude/src/Tool.ts` + `src/tools/*.ts`(V04) | Tool 的声明结构(name / description / schema / handler 如何组织)。我们的 10 个 render tool 严格按这套模式一个文件一个 tool |
| **§ 5.3 `guidance.md` · 教 Agent 何时用哪个工具** | Claude Code 的 **CLAUDE.md 系统提示 + Skill 的 SKILL.md frontmatter**(V05)| Claude Code 怎么用精简文字告诉模型"什么场景用什么工具" —— 这是 prompt 工程的核心。**直接参考 Claude Code 官方 built-in skill 的 SKILL.md 写法**,口吻、段落结构、判断表都可借鉴 |
| **§ 6 前端可插拔渲染** | Ink 的 custom renderer(V01)| 只抽"按 component 名查表渲染"的机制。具体组件长相走 `product/03-visual-design.md`(Linear Precise),**不照搬 Ink 的外观** |
| **prompt description 字段怎么写** | Claude Code tools 的 `description` 字段(V04 所举例) | 描述语言要**告诉模型"什么场景用这个,什么场景别用"**,不是单纯介绍"这个工具是什么"。我们每个 `allhands.render.*` 的 description 照这个语气 |

**特别注意**:Claude Code 的 prompt 设计非常精到 —— "工具描述 = 半个 prompt"。autopilot 写 10 个 render tool 的 description 时,**每条都去抽 Claude Code 里同类工具的描述写法**,再适配。
