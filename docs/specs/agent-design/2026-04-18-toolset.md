## 1 · 问题陈述

今天 allhands 的 "工具库" 散:
- `plan_*` 4 个 tool 在 agent-design spec,**只是"agent 对 todo 的声明"**,用户端没法看 / 改 / 续 plan

## 2 · 原则

### 2.1 Tool 即能力边界 · 显式即安全

员工拥有 `skill_ids` → 其中每条 skill 展开成若干 `tool_ids` → **这就是员工能做的全部事**。新装一个 skill = 新 tool 可调用 · 老员工立刻拿到。**不增加代码**。

### 2.2 Claude Code 是参考 · 不是照抄

抄 shape(input_schema · description 写法 · scope 语义 · return shape)· 不照搬实现(Claude Code 是桌面工具 · 我们是 self-hosted 平台,沙箱 / 权限模型得重做)。

### 2.3 AG-UI 流式协议 · 单一事实

每个 tool 从 **发起** 到 **结束** 在 SSE 上走同一组事件(§ 10)。前端组件只看事件 kind + tool_id · 不 switch case 每种 tool · `component-registry` 查表决定渲染。

### 2.4 安全默认保守

- Sandbox 工具族(命令执行 / 代码运行 / FS write)scope 默认 WRITE · `requires_confirmation=True`(可被 workspace policy 关掉)
- Web 抓取 READ · 但 domain allowlist / blocklist 可配
- 每个 tool 带 `preview`(ReadOnly 的 "我会干啥")—— 给用户 confirmation 面板看

### 2.5 不重复设计,只对齐

- Plan → **扩展** agent-design § 5(§ 4 本 spec 保持兼容,加新 tool)
- SubAgent → **合并** agent-design § 6 dispatch + tasks.create(§ 5 本 spec 给统一命名空间 · 不废旧 tool)
- 其他族全是新增

## 6 · Sandbox 族

### 6.1 架构

docker-compose 加新 service `allhands-sandbox`:
- 基础镜像:`python:3.12-slim` + `node:20-alpine` 多阶段(轻)· 默认进 bash
- 每个 conversation / task 起一个 **session 容器**(懒启 · idle 10 min 回收)
- Backend 通过 Docker SDK 连接 · exec_run 发指令
- 文件系统:容器内 `/workspace` 挂载 `backend/data/sandbox/{session_id}/`(宿主机可见 · artifact 整合点)
- 网络:默认 `--network=sandbox-net`(自定 bridge · DNS 允许 allowlist · 其他块)· 可通过 policy 开/关
- 资源:`--memory=1g --cpus=1 --pids-limit=256`

### 6.2 Tool

| Tool id | scope | params | 返回 | confirmation |
|---|---|---|---|---|
| `sandbox.exec` | WRITE(若 cmd 在禁名单)· IRREVERSIBLE 级 | cmd(str)· cwd? · timeout_s=60 · background=false · network=false | {stdout, stderr, exit_code, duration_ms, job_id?} | 第一次 per session · 后续 sticky(同 session 同类 cmd 不再问)|
| `sandbox.run_script` | WRITE | lang(python/node/bash)· code(str)· timeout_s=60 · background=false | 同上 | 同上 |
| `sandbox.read_output` | READ | job_id · offset=0 | {stdout_chunk, stderr_chunk, done, exit_code?} | 否 |
| `sandbox.kill` | IRREVERSIBLE | job_id | {killed: bool} | 是 |
| `sandbox.list_jobs` | READ | - | [{job_id, cmd, status, started_at}] | 否 |

### 6.3 安全约束(强制 · 一个都不能跳)

1. **禁名单命令**(regex 黑):`rm -rf /`, `dd if=`, `:(){ :|:& };:`, `mkfs`, `mount`, `iptables`, `nc -l` · 检测到 → 拒绝 + emit `sandbox.blocked` 事件到 cockpit
2. **容器逃逸防护**:`--privileged=false --cap-drop=ALL` · 只加 `--cap-add=DAC_READ_SEARCH`(如确需读挂载文件)
3. **默认断网**:`network=true` 要用户 confirmation · 开了也走 sandbox-net(DNS allowlist 默认 `pypi.org, npmjs.org, registry.npmjs.org, github.com, *.githubusercontent.com`)
4. **超时硬 kill**:timeout_s 到了 · 容器级 SIGTERM 后 SIGKILL
5. **Output 体量**:stdout/stderr 超过 1MB · 截断 + 存到 artifact 让用户 "完整看" (不阻塞 agent)

### 6.4 Sticky confirmation

首次 `sandbox.exec` 出 confirmation 面板:

```
allhands 想在沙箱里执行:
  $ ls -la /workspace

选项:
[ ] 允许 · 后续 session 内同类 cmd 不再问
[ ] 允许 · 每次都问
[ ] 拒绝
```

"同类 cmd" = 首个 token 相同(ls / git / python / pip ...)+ 无 network 开关。

### 6.5 前端渲染 · `SandboxTerminalCard`

- 顶:`$ {cmd}` 灰色 prompt + 时长 · exit code 彩色(0 绿 / ≠0 红)
- stdout 白 mono · stderr 红 mono
- 长输出折叠(首 50 行 + "展开全部 + 下载")
- 实时流(AG-UI tool_result.delta)· 光标闪
- 右上角按钮:`复制命令` · `重跑` · `kill`(run_in_background 时)· `看完整 output`

### 6.6 参考源码

| 本节 | ref-src-claude | 抽什么 |
|---|---|---|
| sandbox.exec 含义 / description | **Bash** tool(V04 Bash 子章) | 三段式 description · 指导 agent "何时 bash / 何时其他 tool" · timeout 约定 · background 约定 |
| Sticky confirmation | Claude Code 的 permission 记忆(V04 末段 · permission modes) | 已批准的 pattern 在同 session 不重复问的设计 |
| 长输出截断 + `sandbox.read_output` | Claude Code 的 BashOutput / Monitor tool(V04 或 V0N) | 长任务持续拉 output 的 idiom · 封装成独立 tool |
| 代码执行 `run_script` | Claude Code 的 Python/JS one-shot(V04 若有) | code block 直传 + lang hint |

## 8 · FS 族(受控 workspace)

### 8.1 作用域

**只能**读写 `backend/data/sandbox/{session_id}/`(与 sandbox 同目录)+ `backend/data/artifacts/{workspace_id}/`(artifact 存储)两个白名单路径。任何路径 escape(`..` / 绝对路径非白名单) → 拒绝。

### 8.2 Tool

| Tool id | scope | params | 返回 | 其他 |
|---|---|---|---|---|
| `fs.read` | READ | path · offset? · limit? | {content, total_lines} | Claude Code Read 的 shape |
| `fs.read_many` | READ | paths[] | [{path, content}] | 批量优化 |
| `fs.write` | WRITE + conf | path · content | {bytes_written} | 已存在 → 要 confirmation |
| `fs.edit` | WRITE + conf | path · old_string · new_string · replace_all? | {replacements} | Claude Code Edit 的 shape · 报 preview diff |
| `fs.glob` | READ | pattern · path? | [paths] | Claude Code Glob |
| `fs.grep` | READ | pattern · path? · glob? · type? · output_mode? | 按 mode | Claude Code Grep · 支持 content/files/count 三 mode |
| `fs.delete` | IRREVERSIBLE + conf | path | {deleted: bool} | - |

### 8.3 diff 预览(fs.edit 的核心 UX)

`fs.edit` 在 confirmation 面板显示完整 unified diff(`MarkdownCard` 里 ```diff 块)· 用户看到底改哪几行才批 · 不是盲批。

### 8.4 前端渲染

- `fs.read` → `FilePreview`(shiki syntax highlight · 行号 · 可复制)
- `fs.glob` → `MatchList`(每行一个 path · hover 显示 size / mtime · 点击展开预览)
- `fs.grep` → 按 output_mode:content 时 grouped by file · files/count 时简表
- `fs.write` / `fs.edit` 成功后 → 紧跟一个 `FileDiffCard` 显示实际改了什么(不是 confirmation 里的预览 · 是 result)
- `fs.delete` → 小红 `FileDeletedCard`("`path` · <size> · `撤销` (v1)")

### 8.5 参考源码

## 9 · Think 族 · `think.scratch`

### 9.1 为什么

Claude Code 的 agent 有"内省"(ExtendedThinking / scratchpad)习惯 —— 长决策前写几句自问自答。本 spec 给一个**无副作用**的 think tool:记录思考 · **不改任何状态** · 前端用小折叠块展示。

### 9.2 Tool

```python
Tool(
    id="allhands.meta.think.scratch",
    scope=ToolScope.NONE,      # 新增 scope 枚举值(与 READ/WRITE/IRREVERSIBLE/BOOTSTRAP 并列)
    requires_confirmation=False,
    description="""Record a short internal thought for yourself before making a decision.
Use BEFORE calling a non-trivial tool (dispatch, edit, delete, exec) to:
  - list what you know
  - list what you don't know
  - pick the narrowest next step

Don't use for: explaining an answer back to the user (just answer), or
restating the question (wastes tokens).

Arg: text (short · < 200 words).""",
)
```

### 9.3 前端渲染

`ThinkBubble`:灰字斜体 · 默认 **折叠**("💭 thought · 3 lines · 展开")· 不打断阅读节奏。聊天密度考虑:**连续 3 个 think 自动合并** 成一个折叠区。

### 9.4 参考源码

| 本节 | ref-src-claude | 抽什么 |
|---|---|---|
| think.scratch | Claude Code 的 ExtendedThinking / internal reasoning pattern(V02 query.ts · thinking stream) | 分离 "思考" 与 "输出" · 思考走独立 stream kind · 前端默认折叠 |

## 10 · AG-UI 流式渲染契约

### 10.1 SSE 事件 kind(本 spec 规范 · 前后端强一致)

| event kind | payload schema | 触发时机 |
|---|---|---|
| `message.start` | {conversation_id, role} | assistant turn 开始 |
| `message.delta` | {text_chunk} | assistant 文本 token 流 |
| `message.end` | {finish_reason} | assistant turn 结束 |
| `tool_call.start` | {tool_call_id, tool_id, name} | tool 被调 · 还没填参数 |
| `tool_call.args_delta` | {tool_call_id, args_json_chunk} | 参数流式填入(model 支持则用)|
| `tool_call.args_complete` | {tool_call_id, args} | 参数全了 |
| `tool_result.start` | {tool_call_id} | 开始执行 |
| `tool_result.delta` | {tool_call_id, stdout_chunk?, stderr_chunk?, data_patch?} | 执行过程增量(sandbox stdout / web stream / think 内容) |
| `tool_result.end` | {tool_call_id, ok, summary, payload_ref?} | 执行完 |
| `render` | {component, props, patch_target?} | 已有的 render event · tool_result 常附带一个 |
| `confirmation.request` | {confirmation_id, tool_id, scope, summary, preview} | Gate 拦截 |
| `confirmation.decision` | {confirmation_id, decision} | 用户回决 |

### 10.2 前端消费模型

`web/lib/agui-stream.ts`:

```ts
class AGUIStream {
  onEvent(handler: (e: AGUIEvent) => void): void
  dispatchTo(messageList: MessageList): void   // 根据 kind 路由到对应 UI state
}
```

组件注册:


每个组件只收 `{ toolCall: ToolCallState, stream?: ReadableStream }`。不 switch 写死。

### 10.3 默认渲染 fallback

未注册的 tool_id → `GenericToolCallCard`(args preview · result preview · 可折叠)· 永远不留白屏。

### 10.4 流式 UX 细节

- **tool_call.start 到 tool_result.end 之间**:卡片显示 spinner · 灰边
- **tool_result.delta**:增量进卡片(stdout 每行 append · image 成型进 ImagePreview)
- **tool_result.end 错**:卡片边框红 · 显示 error · "重试" 按钮(可选)
- **confirmation.request 中**:卡片显示"等你批准" · 顶部 banner 同步出
- 所有过度效果 **遵守 CLAUDE.md § 3.5.3** 克制 · **禁止** Framer Motion · 仅 CSS `--dur-fast` 过度

## 14 · 开放问题 · Decision defaults

1. **Q**: Sandbox 用 Docker 还是更轻的(firejail / bwrap)?
   **Default**: Docker。理由:跨 macOS/Linux 一致 · allhands 已 compose 化 · bwrap 只 Linux。

2. **Q**: Web search provider 选哪家?
   **Default**: v0 `WEB_SEARCH_PROVIDER=tavily`(free tier · API key)· 文档说明用户可切。

3. **Q**: `think.scratch` 真的让 agent 少犯错吗,还是废话?
   **Default**: 装 · 但 prompt 里说 "用之前算算价值 · 用完了的结论直接行动"。Round 3 self-review 看。

4. **Q**: Sandbox 的 `/workspace` 如何 bootstrap 内容?
   **Default**: session 首次 exec 时空目录。用户通过 fs.write 或 artifact.pin 填。clone 本仓库 / 挂用户磁盘 v0 **不做**。

5. **Q**: `fs.*` 和 artifacts-skill 的 `artifacts.create/update` 何时用哪个?
   **Default**:
   - `fs.*` = 原始文件操作(agent 思考过程中)· 不进 artifact 索引
   - `artifacts.*` = **成品**登记(让用户可见 / 下载 / 链)
   - pattern:agent 可能 `fs.write → 自己看 → fs.edit → 确认好了 → artifacts.pin(path)`(artifacts-skill 新 tool · 注册已有文件为 artifact)

6. **Q**: `subagent.dispatch` vs `subagent.create_task` · 只用一个是不是简单?
   **Default**: 两个都留。理由:语义差异显著(同步/异步) · prompt 教用对,不是合并。
