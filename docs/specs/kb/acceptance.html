§3 · UI 验收 · 五件事

浏览器打开 /knowledge · 按下面顺序点。每项一句话能验。

3.1 创建一个 KB · 选择 embedder

左下"新 KB 名称"输入 Personal Brain;下方 Embedding model 下拉应该看到 6 项 — mock 两个(默认,可用)、openai/bailian 各两个(无 key 时 disabled 并显示 reason)。点 + 新建 KB。

期望 KB 出现在左侧列表 · 中部 header 显示 mock:hash-64 · 状态计数 0 docs · 0 chunks · 64d
3.2 上传不同格式各一份

点 + 上传文档 · 依次试 4 种:

  • • 一份 .md(任意 markdown)— 应有 heading 切片
  • • 一份 .txt — 走 plain 路径
  • • 一份 .html(可保存任一网页)— script/style 应被剥
  • • 一份 .pdf(任意 PDF)— 每页变 Section
每份上传后 ~1-3 秒 · 状态 pill 从 indexing 变 ready · chunks 数显示出来。
3.3 Hybrid 检索

在搜索框里输入你刚上传文档里出现的关键词,按 Enter。

右侧出现结果卡 · 每条带 cite(doc id + section + page)、score、BM25 #N、vec #N。出现 RRF 融合后的 top-k。
3.4 检索 Tune 面板

点 ⚙ Tune · 把 BM25 weight 设 0(关掉 BM25),保存。再搜同一个 query。

期望:结果顺序变 — 现在只剩向量这条线;mock embedder 下结果差异肉眼可见(分数变小)。再切回 1.0/1.0 恢复。
3.5 删除 + 软删校验

(可选)调 REST DELETE /api/kb/<id>/documents/<did> 或后续做 UI 的删除按钮。

删除后 list 不再返回该 doc · DB 里看 deleted_at 非空(软删 30 天)

§4 · REST 验收 · 14 个端点

不想点 UI 也能验。复制命令到终端跑。

列 KB · 列模型
curl -s http://localhost:8000/api/kb | jq
curl -s http://localhost:8000/api/kb/embedding-models | jq
建 KB
curl -s -X POST http://localhost:8000/api/kb \
  -H 'content-type: application/json' \
  -d '{"name":"curl-test","embedding_model_ref":"mock:hash-64"}' | jq
上传文档(替换 KB_ID)
echo "# Hello\n\nThis is a hybrid retrieval test." > /tmp/sample.md
curl -s -X POST "http://localhost:8000/api/kb/<KB_ID>/documents" \
  -F "file=@/tmp/sample.md" -F "title=sample" -F "tags=demo" | jq
搜索
curl -s -X POST "http://localhost:8000/api/kb/<KB_ID>/search" \
  -H 'content-type: application/json' \
  -d '{"query":"hybrid retrieval"}' | jq
调检索权重
curl -s -X PATCH "http://localhost:8000/api/kb/<KB_ID>/retrieval-config" \
  -H 'content-type: application/json' \
  -d '{"bm25_weight":0,"vector_weight":1.0,"top_k":5}' | jq
颁发 grant(给某员工写权限)
curl -s -X POST "http://localhost:8000/api/kb/<KB_ID>/grants" \
  -H 'content-type: application/json' \
  -d '{"scope":"write","employee_id":"emp_xxx"}' | jq
完整端点表
方法路径说明
GET/api/kb列 workspace 下的 KB
POST/api/kb建 KB
GET/api/kb/embedding-models列可用 embedding provider
GET/api/kb/{id}取 KB 详情
DELETE/api/kb/{id}软删 KB
PATCH/api/kb/{id}/retrieval-config改检索配置
GET/api/kb/{id}/documents列文档(可 filter)
POST/api/kb/{id}/documents上传文档(multipart)
GET/api/kb/{id}/documents/{did}取文档元数据
GET/api/kb/{id}/documents/{did}/text取文档原文
DELETE/api/kb/{id}/documents/{did}软删文档
POST/api/kb/{id}/searchHybrid 检索
GET/api/kb/{id}/grants列 grants
POST/api/kb/{id}/grants颁发 grant
DELETE/api/kb/{id}/grants/{gid}收回 grant

§5 · Agent 端到端 · kb-researcher

这是最有说服力的一段。Lead 应该能用 KB 回答你。

给 Lead 挂上 kb-researcher skill

/employees → 找 Lead → Skills 里加 allhands.skills.kb_researcher → 保存。

如果没有这条 skill,检查 backend/skills/builtin/kb_researcher/SKILL.yaml 是否存在;启动日志里应有它的注册。
在 /chat 里问 Lead

用你 §3.2 上传文档里出现过的内容提问 · 例如:

我刚上传到 KB 的那份 markdown 里讲了什么?引用具体段落给我。
期望 Lead 调用顺序(在 trace 里能看到):
resolve_skill(allhands.skills.kb_researcher)
→ kb_list() → kb_search(kb_id, query)
→ 必要时 kb_read_document(doc_id) → 答复带 citation。
写权限闭环(可选,验权限护栏)

让 Lead 试 kb_create_document · 默认会拒(没 grant)。

把刚才的回答总结成一篇文档,沉淀到 KB。
期望:第一次返回 {error: "no_grant", message: "Ask the user to call kb_grant_permission first"} · 你说"给它 write 权限"后,Lead 调 kb_grant_permission · Confirmation Gate 弹出 · 你点 Allow · 第二次 kb_create_document 通过。
