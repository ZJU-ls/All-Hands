A 全局 Toast 通知系统 3a2246c

问题
所有"操作成功 / 失败"反馈都靠 alert 或静默——不可见、不可堆叠。
方案
基于 Linear / Vercel / Sonner 调研，落地一个最小可用 toast 系统。
关键文件
web/components/ui/Toast.tsx · 在 AppShell 顶层挂载 ToastProvider。
  • 四种语义：success / error / info / warning，各自默认时长（success 2.4s · info 3s · warning 4s · error 6s）。
  • 位置：底部居中，与 Stream 状态 chip / 批量动作 bar 一致。
  • 单击任意位置关闭，最多堆叠 4 条，旧的从底部丢弃。
  • provider 缺失时自动降级为 noop（避免 storybook / 孤岛渲染崩）。
  • API：const { success, error, info, warning, push, dismiss } = useToast()。

B ConfirmDialog 替换 window.confirm 3a2246c

问题
/artifacts 批量删除走原生 window.confirm，文案僵、与品牌不一致、Esc 行为不可控。
方案
统一用项目内置 ConfirmDialog，附带 toast 反馈结果。

  • per-item try/catch，聚合失败计数，根据 0 / partial / total 三段在 toast 显示对应语义（success / warning / error）。
  • 软删除提示文案："Soft delete · still recoverable from the database. The metadata row stays for 30 days."

C 制品 Pin/Unpin 乐观更新 3a2246c

问题
旧实现等 API 全部完成才刷新 UI，单击批量 Pin 后界面"卡住"几百毫秒。
方案
立即翻转本地 items.pinned，按行做 try/catch，单个失败只回滚那一行。
  • UI 立刻反映新状态，SSE 后续帧只是确认（不再造成二次跳变）。
  • 失败时 toast 用 { failed }/{ total } 句式，副标题写 "Reverted those rows"。
  • 这是 Linear / GitHub / Notion 列表批量操作的标准做法。

J DesignForm 未保存离开警示 48635fe

问题
在编辑员工时关掉 tab 或刷新页面，未保存的修改完全丢失。
方案
新建 useUnsavedWarning(active) 钩子，在 active=true 时挂 beforeunload。
  • 条件：isEdit && isDirty && !busy——保存进行时不再二次警告。
  • 仅 active=true 时挂 listener，cleanup 自动 detach。
  • 其他表单（Skill 编辑等）后续可直接复用同一 hook。

N Toast 路由切换清理 3262a90

问题
在 /artifacts 触发的 success toast 跟着用户跳到 /chat——上下文已经失效。
方案
ToastProvider 监听 usePathname()，每次变化丢掉所有非 error toast 并取消其 timer。
  • error toast 保留：失败可能要重试，新页面可能正是用户去重试的地方。
  • timer 在 dispose 时同步清，避免 setState-after-unmount。
