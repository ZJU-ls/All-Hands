2. 改进前后对比

5 个最显眼的真实修改。每个都是用户能直接感知到的 UX 变化。

Before · meta description 静态英文

// app/layout.tsx
export const metadata: Metadata = {
  title: "allhands",
  description: "One for All — open-source
    digital employee organization platform",
};
无论用户语言偏好,搜索引擎、社交分享卡片、浏览器标签页 description 都只显示英文。

After · 跟随 locale 切换

export async function generateMetadata() {
  const t = await getTranslations("metadata");
  return { title: "allhands",
    description: t("description") };
}
实测:EN cookie → "an open-source, self-hosted digital workforce platform" · ZH cookie → "开源自部署的数字员工组织平台"

Before · split-translation 反模式

{t("firesPrefix")}{" "}
<span>{trigger.fires_total}</span>{" "}
{t("firesSuffix")}
中文 "触发 N 次" 分三段拼接 — 英文 "Fired N times" 词序勉强能套,但任何形容词 / 副词都会破。这是 i18n 标准反模式。

After · 单 ICU 字符串 + rich tag

{t.rich("firesTotal", {
  n: () => <span>{fires_total}</span>
})}
// "Fired <n></n> times" / "触发 <n></n> 次"
翻译者拿到完整句子,可任意挪动占位符位置;rich tag 还原 inline 样式。

Before · 后端错误信息裸 f-string

raise HTTPException(
  status_code=404,
  detail=f"Conversation {id!r} not found.",
)
中文用户拿到的 JSON detail 永远是英文。前端只能靠状态码兜底。

After · t() + Accept-Language

raise HTTPException(
  status_code=404,
  detail=t("errors.not_found.conversation_id",
           id=repr(conversation_id)),
)
# en: "Conversation 'X' not found."
# zh: "会话 'X' 不存在"
后端按请求 Accept-Language 切。前端可直接展示 detail,不用再翻一道。

Before · 边角硬编码中文

function relativeTime(iso: string) {
  if (diff < 60_000) return "刚刚";
  if (diff < 3600_000)
    return `${m} 分钟前`;
  ...
}
ArtifactPeek / ArtifactGrid 的 hover 时间标签;没人查时间格式化的硬编码,英文用户看到中文。

After · helper 接 t 参数

function relativeTime(iso, t) {
  if (diff < 60_000) return t("justNow");
  if (diff < 3600_000)
    return t("minutesAgo", { n: m });
}
// catalog: artifacts.peek.justNow 等
ArtifactPeek 与 ArtifactGrid 共享同一组 keys (DRY) · 助手函数 — 不用 hook,把 t 当 dependency-injection 注入。

Before · CommandPalette hint 全是英文

const ENTRY_DEFS = [
  { key: "chat", hint: "chat with lead", … },
  { key: "skills", hint: "skill packs", … },
  { key: "gateway", hint: "provider + model
    gateway", … },
  ...
];
⌘K 命令面板的 19 个入口副标题写死英文;中文用户读不爽。

After · entries 都走 t()

const ENTRIES = ENTRY_DEFS.map(e => ({
  ...e,
  label: t(`entries.${e.key}`),
  hint: t(`hints.${e.key}`),
}));
// ui.commandPalette.hints.chat = "和 Lead 对话"
// ui.commandPalette.hints.skills = "技能包"
label + hint + 底部 footer 全部 i18n · 中文用户看到 "和 Lead Agent 对话 / 技能包 / 供应商与模型网关"。

4. 已知遗留 / 故意不翻译

不是漏掉,是按 i18n 设计原则故意保留为单语。

  • Agent system prompts · backend execution/prompts/lead_agent.md 等 · 用户 SYSTEM 提示词 — 翻译会改变模型行为,且当前用户群以中文为主。
  • akshare 字段名映射 · backend execution/market/akshare_provider.py · "最新" / "昨收" / "发布时间" 等是 akshare 库返回数据的 Chinese key,必须保留以匹配。
  • BrandMark / PlanCard regex 模式 · /qwen|tongyi|通义/ · /批准|同意/ · 这些 regex 匹配的是用户输入中可能的中文品牌名 / 关键词,模式本身就是 Chinese 字面量。已在 i18n-no-hardcoded-zh.test.ts 显式 allowlist。
  • API 参数名 · temperature / top_p / max_tokens / cron 等 · 跨语言通用的 OpenAI / cron 协议词,翻译反而会让用户找不到对应字段。
  • 友好默认 record 名 · seed_service 里的 "阿里云 百炼" 等是用户首次创建时拿到的 record name,可以随后改名 — 不强制翻译。
  • brand wordmark / 装饰短句 · "v0 · mvp" / "CPT" / "EP" / "M" pill 等 · 设计上故意用 ASCII。
  • 注释 / docstring · 项目 maintainer 全是中文母语,JSDoc 与 Python docstring 维持中文。已在两个 hardcoded scan 测试中自动剥除。

5. 推荐 next steps

  • e2e 视觉回归两个 locale 各跑一次 · 当前 playwright 主要在 zh-CN 跑 · 加一组 npm run test:e2e -- --headed --project=en 用 cookie 切 EN 跑视觉 snapshot · catch 长英文按钮溢出。
  • 添加 ja / es 第三语言 stub · 现在 catalog 形状测试只校验 zh-CN ↔ en 一致 · 加新 locale 时,把 missing_in_X 改成"两两交叉"自动适配。
  • backend prompt i18n 实验 · ai_explainer 现在 Chinese-only · 可以让 system prompt 也按请求 locale 切换("…用清晰、口语化的中文" → "…in clear, conversational English") · 影响模型输出语种,需要 A/B。
  • 把 audit 加入 CI gate · 当前两个 audit 测试已在 pnpm test / pytest 跑 · 在 GitHub Actions 加专门 step pnpm audit-i18n:strict 让形状回归直接 fail PR · 比埋在测试堆里更显眼。
  • 触发器 cron preset 的 timezone · "Every day at 08:00" / "Mondays at 09:00" 默认是 UTC · 中文用户多在 +8 · 可以读用户 settings 的 timezone 渲染。
  • error stream 前缀 · [错误] / [error] 用作 stream 出错时的前缀;前端可以解析这个前缀做样式区分,而不是直接拼字符串展示。
  • 给翻译者一个 maintainer 入口 · 现在加新 key 必须改两个 JSON · 可以做一个简单 web UI 让非技术 contributor 编辑翻译,后端持久化到 messages/。

6. 流程亮点

  • 子 agent 并行 · KB 页两次大改(原 ~500 行 → 1496 行 → 2641 行)各派一个子 agent 做 i18n,主线程同时做 catalog audit 与翻译质量复审。
  • 每轮 ~30 分钟 · pull main → 选 2-3 个领域 → 跑 typecheck/lint/test/build → commit + push main + sync(冲突主线解,从不 force)。
  • 回归网入测 · 4 套测试守护现在与未来 — vitest 形状 audit / vitest no-hardcoded-zh / pytest router 扫描 / pytest catalog 形状对齐。
  • 实战验证 · 不光信单测;Round 4 起每轮都用 pnpm build && next start -p 4001 抓 EN/ZH HTML 比对真实输出。<meta description> 漏掉就是这么发现的。
  • frontmatter 不冲突 · catalog 拆 8 个分领域文件 · 多个 agent 并行各写自己的命名空间不互踩。
