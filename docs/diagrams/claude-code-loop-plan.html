§ 1 5 条设计公理 · 这次重写的根

A1

没有平行状态机

"等外部信号"全部经由 deferred tool 表达。loop body 永远是同一个 while-true。

A2

State 是终态化的

Conversation = messages + mode + repos · 全部可从 DB 重建 · 没有内存里藏的状态。

A3

AG-UI 是适配层

不是内部 SoT。execution/ 内事件按 Claude Code 风格,出 SSE 时翻译成 AG-UI。

A4

Deferred tool 一等公民

"挂起等外部信号 → 唤醒"是显式协议。confirmation/clarification/sub-agent 共用同一段代码。

A5

Tool-as-everything

新需求起手式:先问"这能做成 tool 吗?"。能就做成 tool。这是 §3.1 Tool First 的内化扩展版。

§ 2 Deferred Tool · 一个原语承载四件事

Tool 的 call() 中途挂起、等外部信号才返回。从 agent loop 看,它就是个普通 tool · 只是花了点时间。 Loop 不暂停,只有这个 tool 调用在 await。 

class DeferredTool(Tool):
    async def call(self, args, ctx):
        signal_id = await ctx.publish_request(...)   # 把请求发出去
        result = await ctx.signals.wait(signal_id)   # 等 channel 回执 · 自然 await
        return self.unpack(result)

§ 3 Tool Execution Pipeline · 唯一一条路

所有 tool 执行走同一条流水线。maybe_defer 是统一暂停点。审批 / 问用户 / 等子 agent 等所有"等外部信号"语义都从这里入口。

ToolUseBlock from AssistantMessage
↓ validate schema · scope · mode-allowed
↓ permission_check → Allow(executor) | Defer(signal,...) | Deny(reason)
↓ maybe_defer if Defer: signal.publish() → ConfirmationRequested → signal.wait() → outcome
↓ execute | skip(rejected/expired)
↓ record · ToolMessage(tool_use_id, content)

§ 4 内部事件 · 终态层 + 预览层

TERMINAL 终态事件 · 持久化 · 驱动状态

  • AssistantMessageCommitted(message)
  • ToolMessageCommitted(message)
  • ConfirmationRequested(confirmation_id, ...)
  • LoopExited(reason: completed | max_iterations | aborted | prompt_too_long | stopped_by_hook)

commit 后不可变。services/chat_service.py 直接消费这一层做持久化。

PREVIEW 预览事件 · 临时 · 仅 UX

  • AssistantMessagePartial(message_id, text_delta, reasoning_delta)
  • ToolCallProgress(tool_use_id, args_delta)

掉包 / 重连可丢。仅前端流式渲染用。最终一致性由 terminal 事件保证。

翻译边界 · api/ag_ui_translator.py 把内部事件翻译成 AG-UI wire 事件(TOOL_CALL_START 等)。 execution/ 不直接 import AG-UI 类型 · AG-UI 只在 api/ 出现。— A3 公理

批内并发 read-only

同批所有 tool 用 asyncio.gather 一起跑 · 但 ToolMessageCommitted 按输入顺序 emit 到流里 · 保证 transcript 可重放。

批间串行 write/deferred

每个 write 是自己的批 · 一个一个 await · UX 上同一轮多个写 tool 会顺序弹 confirm 对话框,不会一起弹。

§ 6 终态视角 · 一次完整 turn 的生命

user
POST /messages → SSE open · message saved to MessageRepo as terminal UserMessage
loop iter 1
model.astream → emit AssistantMessagePartial · 流末组装 → emit AssistantMessageCommitted(text + 2 tool_use blocks)
pipeline
partition([tool_use_a (read), tool_use_b (write)]) → batch 1: [a] read-only · batch 2: [b] write
batch 1
a 直接 execute → emit ToolMessageCommitted(tool_use_id=a, content={...})
batch 2
b permission_check → Defer(ConfirmationDeferred) · publish() 写 row → emit ConfirmationRequested · wait() 在轮询
user
前端弹 confirm dialog · POST /resume {decision: approve} → DB row.status = APPROVED
batch 2
wait() 下个 tick 看到 → execute → emit ToolMessageCommitted(tool_use_id=b, content={...})
loop iter 2
把 assistant + tool_results 拼回 messages · 再调 model · 这次没 tool_calls → emit AssistantMessageCommitted(text only)
loop exit
emit LoopExited(reason="completed") → translator → AG-UI RUN_FINISHED → SSE close
