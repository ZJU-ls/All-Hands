为什么要双层兜底

修复前 BUG

前端只在收到 TOOL_CALL_END / TOOL_CALL_RESULT 时才把状态改为 succeeded。 一旦后端没产出这两个事件,ToolCallCard 永远停在 pending, 刷新页面也不会变 — 因为持久化的消息里 status 字段就是 pending。

修复后 FIX

后端兜底 处理"事件流本身就缺 end"的来源问题(主修); 前端兜底 防御未来任何 SSE 中断 / 客户端断网 / 后端漏发的可能性。 两层独立,任何一层先生效都不会留下永久 pending。

关键代码点

backend · runner.py

维护 seen_tool_call_ids(start 时填)与 ended_tool_call_ids(收到 ToolMessage 时填)。 stream 主循环退出后、DoneEvent 之前,对差集每个 id 合成一条 ToolCallEndEvent(status=FAILED)。

web · store.ts finalizeStreaming

把 streamingMessage 搬到 messages 之前, map 一遍 tool_calls,把状态为 pending 或 running 的强制改成 failed + error: "tool_call_dropped"。 awaiting_confirmation 不动(它是合法挂起状态)。
