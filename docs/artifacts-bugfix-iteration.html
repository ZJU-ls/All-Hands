0 · 现场复现

  • 顶部 stats 卡显示 csv · 2
  • 用户点击 csv 类型 filter
  • 过滤面包屑显示 csv × + 2 个制品
  • 但左侧列表完全空白
  • 右侧详情面板还在显示之前选中的 .drawio 文件

这是两个独立的故障一起暴露:csv 类型不渲染 + selectedId 不回收。

1 · 根因分析(三连级联)

A

前端 KIND_ORDER 不全

critical

web/components/artifacts/ArtifactList.tsx 的 KIND_ORDER 只列了 7+1 个 kind(含一个根本不存在的 video),漏掉 csv / xlsx / docx / pdf。

for (const kind of KIND_ORDER) { const items = byKind.get(kind); // csv 永远找不到 bucket if (items?.length > 0) sections.push({ title: kind, items }); } → csv 制品落入 byKind map 但 KIND_ORDER 不遍历,整组被静默吞掉。
B

KINDS array 不全

critical

filter 下拉根本选不出 csv。用户能从 stats 卡点 "csv · 2" 是因为 stats 卡用 setKind(k) 直接传字符串,绕过了下拉选项。

C

selectedId 不回收

critical

useEffect 拉新 list 后只 setItems(next),从不验证 selectedId 是否还在新 list 里。filter 变化后,详情面板继续渲染 stale 的 drawio。

5 · 设计沉淀

三条心智模型

1. 类型穷举的 Record map 是结构性 bug 防线。
Record<ArtifactKind, T> 的 TS 严格穷举让"漏 csv"立刻爆编译错误。但 KIND_ORDER as const 是元组,不是穷举,漏了不会爆 — 只会运行时静默吞数据。任何"按 kind 路由 / 分组"的数据结构,优先用 Record 而不是 array。

2. selectedId / bulkSelected 是"派生 from items"的隐式状态。
items 一变,所有引用 items[i].id 的状态都该重新对齐。把"自洽回收"做进 fetchList 完成的钩子,而不是散落到每个 mutator 里。

3. 空态文案要回答用户"现在为什么是空"。
"暂无制品"是 0 信息文案。GitHub / Linear 的范式:显式列出当前过滤维度 + 引导清空。

三个常见错误(纠正了)

1. 前后端类型不对齐。
前端 ArtifactKind 含 "video" 是死代码 + UI 误导。每次扩枚举,前端要 Record + 后端 enum 双侧改,任一漏会立刻爆。

2. 单一信号一刷。
SSE artifact_changed 只刷 stats 没刷 list,因为最初实现时只想用它"提示数字变化"。但用户期望"看到的就是真的"— 必须同步刷 list。

3. 长内容无 mask。
长 csv / 大 markdown 滚动到底部时,如果底部直接被 toolbar 切断,用户分不清"是真没了"还是"还有更多"。28px fade-bottom 是低成本高信号的修复。
