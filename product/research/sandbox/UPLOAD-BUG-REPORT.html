① 后端 · 上传失败的根因(Bug A)

位置:backend/src/allhands/services/attachment_service.py:93

原代码

resolved_mime = (mime or "").strip().lower() or _guess_mime(safe_name)

看似合理,但浏览器对未知扩展不一定送空 mime 而是送 application/octet-stream(尤其是 Windows / 部分 Linux 桌面)。 "application/octet-stream" 是 truthy → 跳过 _guess_mime 兜底 → 直接走到下面的 allowlist 检查: 

if not _mime_allowed(resolved_mime):
    raise AttachmentServiceError(f"mime {resolved_mime!r} not allowed")

结果:用户上传 readme.md / config.yaml / slides.pptx 都被前端弹"上传失败"。

修复

provided = (mime or "").strip().lower()
if not provided or provided == "application/octet-stream":
    resolved_mime = _guess_mime(safe_name)
else:
    resolved_mime = provided

顺手补 _EXTRA_MIME 表(.md / .yaml / .csv / .toml / .jsonl 等)— Python 的 mimetypes 在某些平台对 .md 返回 None,我们 seed 一份兜底确保走到 allowlist 时是合法 text/* mime。

② 前端 · 失焦 + 不能打字的根因(Bug B)

位置:web/components/chat/InputBar.tsx:554 + web/components/chat/Composer.tsx:128

原代码

// InputBar
<Composer
  ...
  disabled={!canSendWithContent && (value.trim().length > 0 || attachments.length > 0)}
/>

// Composer
const textareaDisabled = disabled || (isStreaming && !keepTextareaEnabledWhileStreaming);
<textarea ... disabled={textareaDisabled} />

当任意一个上传失败 → anyFailed=true → canSendWithContent=false → InputBar 把 disabled=true 透传给 Composer → 整个 textarea 被禁用。 浏览器对 disabled 的输入框直接 blur,后续敲键也吞掉,所以用户感觉「失焦了,打不了字」。

修复

给 Composer 加一个新 prop sendDisabled,只阻断发送,不动 textarea:

// Composer.tsx
sendDisabled?: boolean;  // 新 · 只 gate 发送按钮
const canSend = !isStreaming && !disabled && !sendDisabled && value.trim().length > 0;
// textareaDisabled 不变 · disabled 这个 prop 改回原本"罕见整体禁用"语义

// InputBar.tsx
<Composer
  ...
  sendDisabled={!canSendWithContent && (...)}  // 替换 disabled
/>

效果:上传失败时 send 按钮变灰,但 textarea 仍可输入 / 删除失败 chip,焦点不丢。

③ 前端 · 焦点偷窃次因(Bug C)

位置:web/components/chat/InputBar.tsx:296,305

原代码

if (rejected.length > 0) alert(rejected.join("\n"));

alert() 是浏览器原生模态对话框,会把焦点完全拿走。即便用户敲回车关掉,焦点也不返回 textarea(回到 body)。这放大了 Bug B 的体感。

修复

替换为内联 warning strip · 5 秒自动消失 · 不动焦点:

const [rejections, setRejections] = useState<string[]>([]);

if (rejected.length > 0) {
  setRejections((prev) => [...prev, ...rejected]);
  window.setTimeout(
    () => setRejections((prev) => prev.slice(rejected.length)),
    5000,
  );
}

// 渲染层
{rejections.length > 0 && (
  <div className="border-b border-warning/30 bg-warning-soft px-3 py-2 ...">
    {rejections.map((m, i) => <div key={i}>...</div>)}
  </div>
)}

④ 为什么三个 bug 缠在一起?

同一交互,三层都漏:

  1. 用户拖 / 选 config.yaml 上传
  2. 前端 preflight 通过(浏览器 mime = octet-stream,前端只检 ALLOWED 而 octet-stream 在 file.type 为空时被放行 — 见 web/lib/attachments.ts:72-76)
  3. POST /api/attachments → 后端 Bug A 拒,返回 400 "mime ... not allowed"
  4. 前端 uploadOne catch → status="failed"
  5. Bug B 触发:anyFailed=true → Composer disabled → textarea 整体禁用 → 失焦 + 打字无效
  6. 用户去拖第二个文件想再试,如果命中 preflight 拒(超大或不允许 mime),Bug C alert() 又把仅剩的焦点拿走

三个独立 bug,任意一个都能让用户卡死。一起修才能根治。
