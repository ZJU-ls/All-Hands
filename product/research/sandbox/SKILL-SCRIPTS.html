Skill 脚本执行 · 现状与方案 2026-04-27

用户问:"现在如果 skill 里有脚本需要执行 · 平台支持吗?" · 顺带 vs Claude Code 对照
核心结论: 不能跑。我们能 读 skill 里的 scripts/*.py(read_skill_file tool),也能在 UI 里 编辑 它们(SkillFileEditor),但 没有任何 tool 能执行它们。Claude Code 风格的 skill(anthropics/skills 仓库里 pdf / pptx / xlsx 那种用 Python 真做事的)· 我们当前体系下用不了 —— agent 读了脚本也跑不了。
📖

读 skill 文件

支持

allhands.meta.read_skill_file(skill_id, relative_path) · 沙盒限 install 目录 · 256KB 上限 · UTF-8。能读 SKILL.md / references/ / scripts/ 任意文件。

✏️

编辑 skill 文件

UI 支持

新加的 SkillFileEditor(CodeMirror 6)能在 web UI 编辑 .md / .yaml / .json / .py · 文件存进 skill 目录。

🔧

执行 skill 脚本

缺失

没有任何 tool 能跑脚本。整个 backend 搜不到 asyncio.create_subprocess / subprocess.run。这是当前 Skill 体系最大缺口。

3设计方案 · 加 run_skill_script tool

3.1 Tool 契约(ADR 0021 风格 · 自解释)

RUN_SKILL_SCRIPT_TOOL = Tool(
    id="allhands.meta.run_skill_script",
    kind=ToolKind.META,
    name="run_skill_script",
    description=(
        "Execute a script inside an activated skill's scripts/ directory. "
        "Use AFTER calling resolve_skill. The script's cwd is the skill root, "
        "stdout/stderr are captured (truncated at 1 MB tail), exit code returned. "
        "Python scripts run in the shared ~/.allhands/skills-venv (pre-installed "
        "from the skill's requirements.txt at activation time). "
        "Path must start with 'scripts/'. Default timeout 30s, max 120s."
    ),
    input_schema={
        "type": "object",
        "properties": {
            "skill_id":   {"type": "string", "description": "Activated skill id."},
            "script":     {"type": "string", "description": "Path under scripts/ (e.g. 'scripts/extract.py')"},
            "args":       {"type": "array", "items": {"type": "string"}, "default": []},
            "stdin":      {"type": "string", "description": "Optional stdin payload (≤ 64 KB)."},
            "timeout":    {"type": "integer", "minimum": 1, "maximum": 120, "default": 30},
            "interpreter":{"type": "string", "enum": ["python", "node", "bash"],
                          "description": "Auto-detected from extension if omitted."}
        },
        "required": ["skill_id", "script"],
    },
    output_schema={
        "type": "object",
        "properties": {
            "exit":   {"type": "integer"},
            "stdout": {"type": "string"},
            "stderr": {"type": "string"},
            "duration_ms": {"type": "integer"},
            "killed": {"type": "string", "enum": ["timeout", "oom", "user", null]},
            "error":  {"type": "string"},  // 只在 schema/safety 失败时
        },
    },
    scope=ToolScope.IRREVERSIBLE,
    requires_confirmation=True,         # 走 DeferredSignal · 用户确认
    cost_hint=CostHint(relative="medium"),
)

3.2 安全边界(沙盒 + 守门)

边界怎么守
路径必须在 <skill_dir>/scripts/同 read_skill_file · _safe_resolve 校验 · 拒绝 .. 和 symlink
cwd 锁死 skill 根subprocess.run(cwd=skill_dir) · 脚本能 import 同目录 helpers · 但出不了
环境变量白名单仅 PATH · HOME · ALLHANDS_SKILL_DIR · ALLHANDS_DATA · LANG · LC_ALL · 不传用户 shell env(防 token 泄露)
资源限制timeout 默认 30s 上限 120s · stdout 截断 1 MB tail · stderr 同 · CPU 限 = 单核 · 内存软限(POSIX resource.RLIMIT_AS)
Confirmation Gatescope=IRREVERSIBLE + requires_confirmation=True · 弹窗显示 完整命令 + 解释器 + skill_id · 可"总是允许这个 skill"
Python 解释器用 ~/.allhands/skills-venv/bin/python(从 user PATH 隔离)· 提前装好常用库
错误回 envelope遵 ADR 0021 · 不吐 Python traceback · 走 {error, field, expected, received, hint}

3.3 依赖管理 · 共享 skills-venv

设计: 不给每个 skill 单独 venv(太重 · 装一次几百 MB)· 而是共享一个 ~/.allhands/skills-venv。skill 通过 requirements.txt 声明依赖 · activation 时检查 + 增量 uv pip install(uv 比 pip 快 10x · 重复装秒级缓存)。

# 2. SkillRegistry 检查 skill 的 requirements.txt
# 3. 若有:执行 `uv pip install --quiet -r requirements.txt --target ~/.allhands/skills-venv/site-packages`

# 4. 失败 → 返回 ToolMessage 告诉 agent "依赖装失败 · 用 fallback skill"

3.4 完整调用链路(从对话到执行)

① 用户:"用 pdf skill 把 invoice.pdf 的表单字段都列出来"

② Lead Agent:
   resolve_skill("pdf")
     → SkillRegistry 注入 body 到 runtime
     → 检查 requirements.txt · 触发 uv pip install(缓存 ~ 1s)
     → SkillRuntime.resolved_skills 加 "pdf"

③ Agent 读 SKILL.md body · 看到 "用 scripts/extract_form_structure.py":
   read_skill_file("pdf", "scripts/extract_form_structure.py")
     → 读到 .py 内容 · 看 main() 签名

④ Agent 决定调用:
   run_skill_script(
     skill_id="pdf",
     script="scripts/extract_form_structure.py",
     args=["/Users/me/Downloads/invoice.pdf"],
     timeout=30
   )

⑤ PermissionGate 弹 confirmation:
   "Lead Agent 想跑:
    python scripts/extract_form_structure.py /Users/me/Downloads/invoice.pdf
    (在 skill: pdf 内 · 共享 venv · 30s timeout)"
   [允许] [总是允许 pdf skill] [拒绝]

⑥ 用户允许 → DeferredSignal 唤起 → executor 跑 subprocess
   stdout: '{"fields": ["customer", "amount", ...]}'
   exit: 0

⑦ ToolMessage 回到 agent · agent 解析 JSON · 答复用户。

5跟 PLAN-V2 的关系

这个能力已经在 PLAN-V2 里隐含(Tier 1 local.run_python 能跑任意 Python)· 但 run_skill_script 是更精准的方式:

用 PLAN-V2 的 local.run_python

  • agent read_skill_file 读到 .py 内容
  • 把整个文件内容塞进 local.run_python(code=...)
  • 问题: 大 .py(50 KB)在 LLM context 里炸 token · 改一个参数要重传整个 .py · 编码问题多
  • 能跑 · 但贵 · 不优雅

用 run_skill_script

  • agent 直接 run_skill_script(skill_id, "scripts/x.py", args=[...])
  • .py 内容不进 LLM context · 只传 args(几十字节)
  • 解释器跑磁盘上的 .py · 改完 reload 即可
  • token 省 99% · 工程清爽
结论: run_skill_script 是 local.run_python 之上的"专用版本" —— skill 场景下成本远低、协议清晰。两个 tool 不冲突 · 共存(local 给临时代码 · skill_script 给 skill 内打包脚本)。
