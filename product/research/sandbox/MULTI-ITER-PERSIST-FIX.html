多轮 turn 按顺序落库 · 修 max_iterations 整段消失 + 重载 tool 排序错乱

1 · 现象

现象 A · 整段消失

触顶 max_iterations 后失败 · 刷新页面 · 中间 ~20 轮 tool 调用 + 文本全部不见 · 只剩用户初始 你能做什么 和 Lead Agent 的开场白。

现象 B · 顺序错乱

偶尔能看到最后一段助手文本 · 但 8 个 create_model / list_models 卡片一股脑挂在文本下面 · 而不是当时实际"列模型 → 创建 → 再列 → 再创建"的交错顺序。

2 · 根因

两个症状一个根。chat_service._persist_assistant_reply 是把 SSE 流落库的 tap,它的 buffer / tool_calls_by_id / message_id 跨整个 turn 累加,只在 done / error 时调一次 flush() 写一行 Message:

async def flush() -> None:
    if persisted or not buffer or message_id is None:
        return    # ← 这里
    ...
    msg = Message(
        id=message_id,           # 最后一次迭代的 id
        content="".join(buffer), # 全部迭代文本拼起来
        tool_calls=list(tool_calls_by_id.values()),  # 全部迭代 tool_calls
        ...
    )
    await self._conversations.append_message(msg)

2.1 为什么"整段消失"

触顶 max_iterations 时,AgentLoop 在下一轮 LLM 调用之前就 yield LoopExited 退出。Runner 把它翻成 ErrorEvent + DoneEvent(error),tap 进 error 分支调 flush()。如果这个 turn 全程只发 tool_call(模型很少补文字 · qwen 系列特别明显),buffer 是空的 · 命中 not buffer: return 提前返回 · 整个 turn 一行 Message 都没落。Reload 自然啥都没。

2.2 为什么"顺序错乱"

就算 buffer 有内容,被持久化的也就一行 assistant Message,带一个 content 串和一个 tool_calls[] 平铺数组。前端 MessageBubble 历史路径走 LegacyAgentBody · 渲染顺序固定:

1. message.content           ← 全部文本拼起来
2. message.tool_calls.map()  ← 全部 tool 卡片堆底部
3. message.render_payloads

实时 SSE 之所以正常,是因为前端按事件顺序构造 segments 数组(text/tool_call/render 交错),用 SegmentedBody 渲染。Reload 没有 segments → fallback 到 Legacy → 错乱。

"按顺序存储,按顺序渲染就好吧?" — 用户原话 · 第一性原理

3 · 修复

每次 LLM 迭代产出一行独立的 assistant Message · 每个 tool_call_end 立刻补一行 role=tool · 时间戳严格单调 · reload 顺序 = 实际执行顺序。

3.1 新事件:AssistantCommittedEvent

class AssistantCommittedEvent(BaseModel):
    """Per-iteration boundary signal · one LLM call's full assistant snapshot."""
    kind: Literal["assistant_committed"] = "assistant_committed"
    message_id: str
    text: str = ""
    reasoning: str | None = None
    tool_calls: list[ToolCall] = []

Runner 收到内部 AssistantMessageCommitted 时,从 content_blocks 里把 TextBlock / ReasoningBlock / ToolUseBlock 各自拣出来,打包成事件 · 在原有 ToolCallStartEvent 之前发出。

3.2 chat_service · per-iteration commit

async def commit_iteration(iter_message_id, iter_text, iter_reasoning, iter_tool_calls):
    iteration_seq += 1
    created_at = (first_seen or now()) + timedelta(microseconds=iteration_seq)
    msg = Message(role="assistant", content=iter_text, tool_calls=iter_tool_calls, ...)
    await self._conversations.append_message(msg)
    persisted = True   # turn_lock 走 happy path
    # 同步落 ASSISTANT 事件到 event_repo (ADR 0017)
    # 重置 buffer / tool_calls_by_id / message_id / first_seen 等待下一轮

3.3 chat_service · per-tool commit

async def commit_tool_message(tc: ToolCall):
    iteration_seq += 1
    created_at = now() + timedelta(microseconds=iteration_seq)
    tool_msg = Message(role="tool", tool_call_id=tc.id, content=..., ...)
    await self._conversations.append_message(tool_msg)

关键:iteration_seq 全 turn 单调递增 · 微秒精度叠加 · 即使同一个 event-loop tick 里连续 commit 4 行 · 时间戳依然严格有序。list_messages 用 ORDER BY created_at · reload 顺序 = 写入顺序 = 实时顺序。

3.4 trailing flush 改造

原 flush() 用 persisted 标志双用(防重写 + 表示是否落过)· 新方案拆开:

  • persisted:首次 commit_iteration 即 True · turn_lock.complete_turn 用
  • trailing_flushed:flush() 内部防重写,仅最后一段 partial(client disconnect / mid-stream error)走 flush

结果:happy path 的尾部 flush() 因 buffer 已空提前返回 · partial path 仍能补上一行 interrupted=True 的 Message。
