0 · TL;DR

  • 目标场景:写代码 / 整理文件 / 清理无用文件 / 操作 Excel 数据核对 + 类似批量本地任务。
  • 核心结论:这类场景需要的不是"语言级沙箱"(Pyodide / WASI),而是 带持久化工作区的 OS 级容器隔离 —— 文件系统、网络、进程、资源都要可控,且能挂载用户给定目录。
  • 推荐路线:主线 Docker(rootless 优先)+ 工作区 bind-mount + seccomp/apparmor + 网络策略;云上备选 E2B;长远观察 Firecracker / microVM。
  • MVP 工作量:~ 1.5 - 2 周(单 worker 起 docker、池化、Tool 接口、Excel 工具链镜像、超时/限额、审计日志)。
  • 必须先决策:① 每会话长生命周期容器 vs 每 Tool 一次性容器 ② 工作区在宿主目录 vs 仅容器内 + 文件下载 ③ 网络默认 deny 还是 allow。

1 · 目标与非目标

1.1 目标(必须)

#能力说明
C1文件读写限定到一个"工作区"路径下 · 不能逃逸
C2代码执行Python / Node / Shell · 装常见库(pandas / openpyxl / numpy / requests)
C3Office 操作xlsx / docx / pdf 读写 · 至少 openpyxl + LibreOffice headless
C4网络可控默认按白/黑名单 · 可针对会话开/关
C5资源配额CPU / 内存 / 磁盘 / 单次执行时长
C6持久化同会话跨多次 Tool 调用复用 · uvicorn reload / 崩溃后能恢复
C7可观测每次执行有 stdout/stderr/exit/耗时 · 进 LangFuse trace
C8多租户不同会话 / 不同用户的工作区严格隔离

1.2 非目标(本期不做)

  • 完全防御内核 0day(Docker default 已经够 v0)
  • GPU 工作负载
  • 浏览器自动化(已有 chrome-devtools MCP)
  • Windows 容器

1.3 隐含约束(来自 CLAUDE.md)

  • Tool First:沙箱能力以 Meta Tool / Backend Tool 出现 · 同时 REST 给 UI 用
  • Pure-Function Loop:执行结果走 InternalEvent · 不能在 AgentLoop 里藏状态
  • Layer Isolation:execution/ 调 services/sandbox/,core/ 不许 import docker SDK
  • Confirmation Gate:写文件 / 删除 / 执行 shell 这种 IRREVERSIBLE 必须走 DeferredSignal

2 · 开源 / 商业方案横评

2.1 OS 级容器(主流推荐)

方案隔离强度启动挂载运维备注
Docker / Podman中100ms-1sbind-mount 直接低生态最成熟 · rootless 可接受
gVisor (runsc)高1-2s · 损耗 10-30%OK中Google · 防御力远超 docker default

2.2 AI Agent 专用沙箱(直接对标我们)

方案形态主要卖点自托管适合我们?
E2B (e2b.dev)云 SaaS · Firecracker启动 ~150ms · FS / 终端 / 端口 / 桌面 / 浏览器一站部分云上强候选;自托管要拼 control plane

2.3 进程级 / 语言级(轻量但够不到我们的场景)

方案适用为什么不够
Pyodide / WASI / WebContainer浏览器 / 边缘跑 Python/JS不能跑 LibreOffice · 不能动宿主文件
nsjail / bubblewrap / firejail单命令快速套个 namespace没工作区生命周期管理 · 没池化
macOS sandbox-exec / Linux Landlock进程级文件系统白名单不够防御 · 但适合本地 dev mode 先跑通
chroot / FreeBSD jail老派隔离不阻拦 root 逃逸 · 不限网络/cgroup

2.4 Claude Code 是怎么做的

结论:Claude Code 本身不"沙箱化" —— 它信任本地用户,靠 权限提示 + 显式工具白名单 把控。
  • Permission system:每个 Bash / Edit / Write 调用按 settings.json 的 permissions.allow / ask / deny 三态决定是否弹确认。
  • Sandbox 模式(较新):macOS 用 sandbox-exec profile,Linux 用 Landlock + seccomp,默认拒网络 + 限文件写域 ⇒ 这就是 Bash 工具上 dangerouslyDisableSandbox 的由来。进程级、单命令的轻沙箱,不是容器。
  • Hooks:PreToolUse / PostToolUse / SessionStart 钩子让用户自己拦截 / 改写工具调用。
  • 没有持久化工作区抽象:CWD = 用户启动 CLI 时的目录,不分租户(单用户 CLI)。
  • MCP 工具走自己的进程:server 跑在用户机 / 远端,Claude Code 不接管隔离。

对我们的启发:

  1. 进程级沙箱(sandbox-exec / Landlock)适合"本地 dev 模式" + "Tool 默认更严"的双层防御。
  2. Permission / Hook 模型直接抄 —— 我们已有 ConfirmationGate + DeferredSignal · pre/post hook 接到 sandbox 入口即可。
  3. Claude Code 没解决多租户/持久化 —— 因为它是单用户 CLI;allhands 是组织平台,必须自己设计 workspace 抽象,这部分没现成可抄。

4 · 实现方案(主线 · Docker 自托管)

4.1 架构分层

L05 execution/sandbox/
  ├── runtime.py          # SandboxRuntime ABC(create / exec / fs / destroy)
  ├── docker_runtime.py   # 默认实现:docker SDK
  ├── e2b_runtime.py      # 备选实现:e2b-sdk(后期)
  └── tools/              # 13 个 Tool 的 executor(薄壳 · 调 service)

L06 services/sandbox_service.py
  ├── 池化(预热 N 个 idle 容器)
  ├── 工作区注册(WorkspaceRepo · 持久化到 DB)
  ├── 配额(用户/会话级)
  └── 审计(每次 exec → LangFuse + DB)

L03 persistence/
  ├── models/workspace.py        # id, owner, status, image, created_at, last_used_at, quota_*
  └── models/sandbox_audit.py    # exec 记录

L08/L07 api/
  └── routers/sandbox.py         # REST:列表 / 详情 / 文件浏览 / 下载流(UI 直用)

Layer 检查: core/ 没有 docker import;execution/sandbox/ 不直接把 docker 类型暴露给 services —— 用自定义的 ExecResult / WorkspaceHandle Pydantic DTO。

4.2 容器配置(默认 image)

# allhands/sandbox-base:0.1
FROM python:3.12-slim

RUN apt-get update && apt-get install -y --no-install-recommends \
    libreoffice-calc libreoffice-writer \
    poppler-utils \
    git curl jq ripgrep fd-find \
    nodejs npm \
 && rm -rf /var/lib/apt/lists/*

RUN pip install --no-cache-dir \
    pandas openpyxl xlsxwriter \
    numpy requests httpx beautifulsoup4 lxml \
    pypdf python-docx \
    pillow

RUN useradd -m -u 1000 sandbox
USER sandbox
WORKDIR /workspace

启动参数:

client.containers.run(
    image="allhands/sandbox-base:0.1",
    name=f"ws-{workspace_id}",
    user="1000:1000",
    read_only=True,
    tmpfs={"/tmp": "size=256m"},
    volumes={host_workspace_dir: {"bind": "/workspace", "mode": "rw"}},
    network_mode="sandbox-egress",
    mem_limit="2g",
    nano_cpus=2_000_000_000,
    pids_limit=512,
    cap_drop=["ALL"],
    security_opt=["no-new-privileges:true",
                  "seccomp=/etc/allhands/seccomp.json",
                  "apparmor=allhands-sandbox"],
    detach=True, init=True,
)

4.3 池化 / 生命周期

  • 每会话一个长生命周期容器(默认):exec 进 docker exec · 文件累积 · 30 分钟空闲 → stop · 24h → 镜像化 + 删容器 · 7d → 询问归档
  • 快速池:预热 3-5 个 idle 容器,新会话直接绑(摊薄启动)
  • 容器死 / 宿主重启 → WorkspaceRepo 是 SoT,服务起来按需 re-create container 挂回原 host dir

4.4 工作区落盘策略(关键决策)

选项方式优劣
A · 推荐${ALLHANDS_DATA}/workspaces/<id>/ bind-mount崩溃不丢 · 备份简单 · UI REST 直读宿主有真实文件 · 要清理
B容器内 named volume · 仅 Tool 取出更隔离 · 没"宿主残留"UI 文件浏览要代理 · 容器删 = 数据没

→ 选 A(allhands 自部署 · 用户本就拥有宿主)。ALLHANDS_DATA 走环境变量,默认 ~/.allhands/data。

4.5 网络策略

  • 自建 docker bridge sandbox-egress
  • iptables 默认 DROP · ALLOW 列表通过 WorkspaceQuota.allowed_domains(默认含 pypi / npmjs / github)
  • DNS 走自托管 dnsmasq + ipset 解析白名单 → 自动放行 IP
  • UI 切换"开放公网"开关 → Meta Tool 改 quota,WRITE + Confirmation

4.6 与 AgentLoop 的对接(ADR 0018 兼容)

  • Tool executor async def,调 services.sandbox_service.exec(...)
  • 长 exec(> 5s)走 ToolCallProgress 内部事件流 stdout 给前端
  • exec 失败 / 超时 → tool 返回 {ok: false, exit, stdout, stderr, killed_reason},不抛异常进 loop
  • IRREVERSIBLE 走 ConfirmationGate · 审批 payload 含命令 + 工作区 ID + 预估资源

5 · 技术选型 · 依赖明细

5.1 Python 端

依赖用途备注
docker (SDK)起容器 / exec主线 ✅
aiodockerasync 版可选 · docker 也能在线程池里跑
aiofiles工作区文件流式读写加
e2b备选 runtime后期

5.2 镜像里的工具栈

  • 必装:python3.12 · node20 · git · ripgrep · jq · curl · libreoffice · poppler · pandas/openpyxl/numpy/requests/httpx/pypdf/python-docx/pillow
  • 可选(动态 image variant):tex-live · ffmpeg · playwright · postgres-client

5.3 隔离层

  • MVP:Docker rootless + seccomp default + AppArmor + cap_drop=ALL + read_only rootfs + tmpfs
  • 升级位:把 docker runtime 换成 runsc(gVisor),一行 daemon.json,代码零改
  • 极端模式:同一 SandboxRuntime ABC 实现一份 firecracker driver(借鉴 microsandbox / firecracker-containerd)

5.4 不引入

  • ❌ Kubernetes(单机起步 · k8s 是包袱)
  • ❌ docker-compose 来管 sandbox 容器(要程序化生命周期 · 直接用 SDK)
  • ❌ E2B 作为唯一方案(自部署哲学冲突)
  • ❌ 自己写 namespace/cgroup wrapper(轮子已经够好)

7 · 风险与未决问题

#风险缓解
R1rootless docker 在某些 Linux(老内核 / 缺 cgroup v2)起不来文档给最低内核 5.13 + docker-rootful 兜底脚本
R2LibreOffice headless 并发偶发卡死单 ws 串行 · 60s 超时 · 严重时切 gnumeric
R3工作区膨胀 / 用户传入大文件默认 quota 5GB · 上传 > 100MB 走 confirmation
R4容器 escape 0day升级路径在 §5.3 第二档 · v0 接受残余风险并文档明示
R5多用户并发资源争抢services 层加全局并发 semaphore + per-user quota
R6macOS / Windows 开发机怎么跑?macOS 用 Docker Desktop · 本地 dev 也支持 sandbox-exec adapter 跑无 docker 模式 · CI 全套只在 Linux

9 · 给评审者的 3 个问题

  1. 自部署哲学下,E2B 这种 SaaS runtime 是否作为可选项接进来(允许用户在 UI 里切到"云沙箱")?还是禁掉只走自托管?
