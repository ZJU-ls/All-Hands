# Plan 0001 · MVP v0

**Status:** Draft
**Owner:** TBD

## 1. Goal

Deliver the "v0 Demo" described in `product/01-prd.md §3` on a single laptop within 5 minutes
using < $0.50 of model spend:

> User talks to Lead Agent → Lead Agent creates a `Researcher` and a `Writer`
> employee (via Meta Tools) → dispatches them → returns a markdown report
> comparing LangGraph and CrewAI.

**Done when:** `docker compose up` works on a clean machine, the user can run the
demo end-to-end, every tool call is traced in LangFuse, and WRITE-scope tools
present a confirmation dialog that the user must approve before execution.

## 2. Non-Goals (v0)

Defer to v1+ per `product/05-roadmap.md`:

- Triggers (event + cron)
- "驾驶舱" dashboard
- Gateway admin UI
- Tool / Skill / MCP marketplace
- Multi-user / RBAC
- Supervisor-mode employees (unified React Agent only; supervisor behavior is
  expressed as tool packages on Lead Agent)

## 3. Done criteria

Objective gates — a PR that claims "v0 complete" must cleanly pass all of:

1. `./scripts/check.sh` green (ruff + format + mypy strict + import-linter + pytest + web lint/typecheck/test)
2. `docker compose up` from a fresh clone reaches:
   - backend `GET /api/health` → 200 OK
   - web `/` renders the "backend ok" badge
   - LangFuse UI reachable at `:3001`
3. Demo script (appendix A) produces a markdown report and every step is visible
   in LangFuse as a single trace tree.
4. Every WRITE/IRREVERSIBLE/BOOTSTRAP tool call was blocked until a
   `ConfirmationGate` decision.
5. Import-linter: `core/` has zero framework imports. CI would fail if that
   regressed.

## 4. Work breakdown

Tasks are ordered for minimum-merge-conflict flow. Each task is self-contained
and shippable. Tag each PR with `[plan-0001/Tn]`.

### T1 · Bootstrap (DONE in this scaffold)

- Monorepo layout, CLAUDE.md, `.claude/`, product docs, ADRs
- `backend/` config (pyproject, ruff, mypy, import-linter, alembic)
- `web/` config (Next.js 15, Tailwind, Vitest, component-registry)
- `core/` Pydantic domain models (Tool, Skill, MCPServer, Employee, Conversation, Message, ToolCall, Confirmation, errors)
- `persistence/orm` SQLAlchemy row types + alembic 0001 initial migration
- `api/` FastAPI app factory + `/api/health` + CORS
- Smoke tests: core models, import-linter layering, health endpoint
- `docker-compose.yml` (backend + web + langfuse + postgres)
- `scripts/check.sh`, `scripts/dev.sh`

### T2 · Repositories (persistence → core mapping)

Implement `allhands.persistence.repositories` concrete classes using the ORM rows.

- `SqlEmployeeRepo`, `SqlConversationRepo`, `SqlSkillRepo`, `SqlMCPServerRepo`,
  `SqlConfirmationRepo`
- Round-trip mapping helpers (`row → domain`, `domain → row`)
- Integration test: insert an Employee, read back, assert equality
- Enforce `is_lead_agent` singleton via DB partial index + service-layer check
- TDD: write repo tests first against an in-memory SQLite

### T3 · Tool Registry + builtin tools

- `execution/tools/` package with `ToolRegistry` (dict + `register()` + `resolve()`)
- `discover_builtin_tools()` returns:
  - `allhands.builtin.fetch_url` — Backend, READ
  - `allhands.builtin.write_file` — Backend, WRITE (demo output of markdown)
  - `allhands.meta.create_employee` — Meta, WRITE
  - `allhands.meta.list_employees` — Meta, READ
  - `allhands.meta.dispatch_employee` — Meta, WRITE (invokes child run)
  - `allhands.render.markdown_card` — Render, READ
- Every tool has: unit test + schema validation (input_schema + output_schema)
- Tools declared as frozen Pydantic `Tool` instances with executor functions
  attached at registration time

### T4 · Skill Registry + seeded skills

- `execution/skills.py` `SkillRegistry`
- Seed Skills: `sk_research` (fetch_url + markdown_card), `sk_write`
  (write_file + markdown_card)
- Skill → expanded tool-id list service helper
- Employee invariant enforcement: `(tool_ids ∪ expanded(skill_ids))` non-empty

### T5 · Agent Runner (LangGraph)

- `execution/runner.py` `AgentRunner` wrapping `langgraph.prebuilt.create_react_agent`
- Checkpointer: `AsyncSqliteSaver` at `settings.checkpoint_db_path`
- Accepts `Employee` + input → yields `SSEEvent` stream
- Integrates `ConfirmationGate` (T6) before executing WRITE+ tools
- Emits LangFuse spans via `get_tracer()` (LangFuse impl lands in T9)
- Unit test: in-process fake LLM returns a tool call, runner streams expected events

### T6 · ConfirmationGate

- `execution/gate.py` `ConfirmationGate` with `request()` / `resolve()` methods
- Persistence via `SqlConfirmationRepo`
- Expiry: `settings.confirmation_ttl_seconds` (default 300s)
- Test plan matrix: scope × requires_confirmation → (bypass | wait | reject)
- `AutoApprovePolicy` test helper for non-confirmation test paths

### T7 · Services layer

- `services/employee_service.py`: create / list / update / delete employees,
  enforces invariants (name regex, `has_any_capability` after skill expansion,
  singleton lead agent, core-meta-tools present if lead agent)
- `services/chat_service.py`: create conversation, send message, stream run
  (returns async generator of SSEEvent)
- `services/confirmation_service.py`: approve / reject / list pending
- Services receive/return `core/` domain objects only — no FastAPI or LangGraph types

### T8 · API transport

- `api/routers/chat.py`: `POST /api/conversations` (create), `POST
  /api/conversations/{id}/messages` (send), `GET
  /api/conversations/{id}/events` (SSE via `sse-starlette`)
- `api/routers/confirmations.py`: `GET /api/confirmations`, `POST
  /api/confirmations/{id}/approve`, `.../reject`
- `api/protocol.py`: pydantic schemas for SSE events, kept in lockstep with
  `web/lib/protocol.ts` (integration test asserts parity)
- Error translation: `DomainError` subclasses → HTTP 4xx with consistent JSON shape

### T9 · Observability wiring

- Replace `_NoopTracer` with `LangfuseTracer` when `langfuse_public_key` is set
- Every Agent Runner span annotated with `conversation_id`, `message_id`,
  `tool_id`, `parent_run_id`
- Confirmation lifecycle events emitted as LangFuse events
- Manual smoke: run demo, open LangFuse, confirm trace tree

### T10 · MCP client

- `execution/mcp_client.py`: stdio/sse/http transport adapters using `mcp` SDK
- Handshake registers server's tools into `ToolRegistry` at runtime
- Health-check loop updates `MCPServer.health`
- Integration test against a canned local stdio MCP server

### T11 · Frontend — Lead Agent chat UI

- `app/chat/page.tsx`: left sidebar (conversation list), main chat pane, right
  confirmation panel
- SSE consumer using native `EventSource` polyfilled via `fetch` streaming for
  POST-with-body
- Components: `MessageList`, `MessageBubble` (role-colored), `ToolCallCard`,
  `ConfirmationDialog`, `RenderSlot` (calls ComponentRegistry)
- State: Zustand store for active conversation + open confirmations
- Vitest: ComponentRegistry dispatch, ConfirmationDialog approve/reject

### T12 · Frontend — Render components

- Minimal `MarkdownCard` component registered against `render.markdown_card`
- Sanitize markdown via `marked` + DOMPurify equivalent; no raw HTML
- Test: given a `RenderPayload`, the correct component mounts; unknown
  component renders a "missing" placeholder

### T13 · Demo script + docs

- `docs/DEMO.md` copy-pasteable prompt + expected report outline
- README "Quickstart" updated with exact `OPENAI_API_KEY=... docker compose up` flow
- Add a CI job that runs `./scripts/check.sh` on every PR

## 5. Risks & open questions

1. **SQLite + concurrent async writes.** Mitigation in ADR 0002: `workers=1`,
   WAL mode, single-writer service layer. If this hurts demo feel, revisit before
   v1 (Postgres option or queue serialization).
2. **LangGraph checkpoint DB path conflict.** Keep it on a separate file
   (`checkpoints.db`) so app DB migrations don't collide.
3. **Confirmation UX latency.** User has to click Approve before tool runs; if
   the chat pane doesn't surface the dialog prominently, the demo feels broken.
   Ship T11 with explicit visual emphasis (toast + highlighted side panel).
4. **LangFuse self-hosting on first run.** Requires manual user creation inside
   LangFuse to get keys. Document this in README under "first-run gotchas".
5. **Protocol drift between `protocol.py` and `protocol.ts`.** T8's integration
   test MUST fail if they diverge — not optional.

## 6. Out-of-scope guards (reject in PR review)

PRs adding any of these get rejected unless a new ADR lands first:

- `mode` field on Employee or Conversation
- REST CRUD endpoint for employees / skills / MCP servers (must go through Meta Tools)
- A Next.js page for employee CRUD (must go through Lead Agent + render tool)
- Framework imports in `core/`
- Tool without `scope` declaration
- Bypassing `ConfirmationGate` outside of test fixtures

## Appendix A · Demo script

```
User: 帮我调研 LangGraph 和 CrewAI 的对比,产出一份 markdown 报告。

Expected Lead Agent behavior:
  1. Plan: "I'll create two employees and dispatch them in parallel."
  2. Tool call: meta.create_employee(name="Researcher", skills=["sk_research"]) → CONFIRM
  3. Tool call: meta.create_employee(name="Writer", skills=["sk_write"]) → CONFIRM
  4. Tool call: meta.dispatch_employee(Researcher, task="compare LangGraph vs CrewAI") → CONFIRM
  5. (Researcher runs: fetch_url ×N, summary)
  6. Tool call: meta.dispatch_employee(Writer, task="write markdown report using Researcher's findings") → CONFIRM
  7. (Writer runs: write_file → CONFIRM)
  8. Render tool: render.markdown_card(report.md) → shown in chat
  9. Final message: "Done — report saved to /data/reports/langgraph-vs-crewai.md"

LangFuse: single trace with nested spans for every tool call, confirmation, and child run.
```
